# PinchTab Agent Benchmark Script

A structured execution plan for AI agents to benchmark PinchTab capabilities through real-world browser tasks.

---

## Token & Cost Tracking

Every step must track token usage to measure real-world agent costs.

### What to Track

For each step, record:
- **Input tokens**: Tokens sent to the model (prompt + context)
- **Output tokens**: Tokens generated by the model (response)
- **Tool calls**: Number of tool invocations
- **PinchTab output size**: Characters/bytes returned by PinchTab commands

### How to Track

**Option A: Model API tracking**
If your model provider exposes usage in responses, capture `prompt_tokens` and `completion_tokens` per step.

**Option B: Estimation**
Use these approximations:
- 1 token ≈ 4 characters (English text)
- 1 token ≈ 3 characters (code/structured output)

**Option C: Session status (OpenClaw)**
If running under OpenClaw, call `📊 session_status` periodically to get cumulative usage.

### Cost Calculation

Use these per-1M-token rates (adjust for your provider):

| Model | Input $/1M | Output $/1M |
|-------|-----------|-------------|
| Claude Haiku | $0.25 | $1.25 |
| Claude Sonnet | $3.00 | $15.00 |
| Claude Opus | $15.00 | $75.00 |
| GPT-4o | $2.50 | $10.00 |
| GPT-4o-mini | $0.15 | $0.60 |
| Gemini 1.5 Pro | $1.25 | $5.00 |
| Gemini 1.5 Flash | $0.075 | $0.30 |

Formula:
```
cost = (input_tokens / 1_000_000 * input_rate) + (output_tokens / 1_000_000 * output_rate)
```

### Recording Format (Extended)

```
Step X.Y: [PASS/FAIL/SLOW]
- Duration: Xms
- Input tokens: X
- Output tokens: X  
- PinchTab output: X chars
- Tool calls: X
- Est. cost: $X.XXXX
- Notes: (any observations)
- Error: (if failed)
```

## Rules

### Execution Rules

1. **Follow steps sequentially.** Do not skip steps unless explicitly marked as optional.
2. **Record outcomes.** After each step, note: success/failure, time taken, any errors encountered.
3. **Stop on blockers.** If a step fails and is marked `[BLOCKING]`, halt the group and record the failure.
4. **Retry once.** If a step fails, retry exactly once before marking as failed.
5. **Fresh state per group.** Each task group assumes no prior state unless it explicitly references a previous group.
6. **Use CLI first.** Prefer `pinchtab` CLI commands. Use HTTP API only when the step specifies it.
7. **Capture evidence.** When a step says "verify", use `pinchtab text` or `pinchtab snap -i -c` to confirm.

### Timing Rules

1. **Wait for navigation.** Use `--wait-nav` when clicking links that navigate.
2. **Re-snapshot after changes.** Always get fresh refs after any click, submit, or navigation.
3. **Timeout threshold.** If any command takes longer than 30 seconds, mark as slow and continue.

### Recording Format

For each step, record:
```
Step X.Y: [PASS/FAIL/SLOW]
- Duration: Xms
- Notes: (any observations)
- Error: (if failed)
```

## Pre-Flight Checks

Before starting the benchmark, verify the environment is ready.

### P.1 Verify PinchTab is installed
Run `pinchtab version` and confirm it returns a version string.

### P.2 Start the PinchTab server
Run `pinchtab server` in background or verify it's already running.

### P.3 Verify server health
Run `pinchtab health` and confirm status is healthy.

### P.4 Verify Chrome is available
Run `pinchtab instances` to confirm no errors about missing Chrome.

### P.5 Clean slate
Stop any running instances with `pinchtab instance stop --all` if needed.

## Benchmark Scope

This benchmark covers:
- **Navigation**: forward, back, reload, block-images, block-ads
- **Observation**: snap, text, diff, find (semantic)
- **Interaction**: click, fill, type, press, select, hover, scroll
- **Tabs and instances**: multi-tab, multi-instance isolation
